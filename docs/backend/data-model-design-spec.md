# 后端数据模型设计规范

本文档规定兰庭流控平台（Lanting Stream）后端数据库表与实体模型的设计原则，所有新增表与实体均应遵循。

## 1. 通用字段约定

每张业务表必须包含以下字段：

| 字段名       | 类型    | 是否可空 | 默认值 | 说明                                                         |
| ------------ | ------- | -------- | ------ | ------------------------------------------------------------ |
| `id`         | INTEGER | NOT NULL | -      | 主键，自增（SQLite `AUTOINCREMENT`）                         |
| `is_delete`  | INTEGER | NOT NULL | 0      | 软删除标志：0 表示未删除；非零（记录 id）表示已删除          |
| `create_time`| INTEGER | NOT NULL | 0      | 秒级时间戳，对应 Java `Long`                                 |
| `update_time`| INTEGER | NOT NULL | 0      | 秒级时间戳，对应 Java `Long`                                 |

- 时间字段统一使用**秒级时间戳**（INTEGER），便于跨时区存储与比较。
- 软删除由 MyBatis-Plus `@TableLogic` 管理，删除时自动将 `is_delete` 更新为当前记录 `id`。

## 2. 命名规范

### 2.1 表名

- 统一使用 `lanting_` 前缀，全部小写，单词间用下划线连接。
- 示例：`lanting_user`、`lanting_workspace`、`lanting_cluster`。

### 2.2 字段名

- 全部小写，单词间用下划线连接。
- 避免使用数据库保留字。
- 枚举类型字段使用 `VARCHAR` 存储，命名以 `status`、`type`、`source` 等后缀结尾。

### 2.3 实体类

- 与表名对应，采用大驼峰命名，去掉 `lanting_` 前缀。
- 示例：`User`、`Workspace`、`Cluster`。

## 3. 软删除与唯一约束

### 3.1 核心原则

**业务名称类字段若需要唯一，必须配合软删除字段使用联合唯一索引 `(name, is_delete)`。**

原因：
- 软删除的记录应视为“已不可用”，但不应永久占用业务名称。
- 若使用单列 `UNIQUE(name)`，删除后同名记录将永远无法再次创建，违背软删除语义。
- MyBatis-Plus `@TableLogic` 删除时把 `is_delete` 设为记录 `id`，每条已删除记录的 `is_delete` 值天然不同，因此 `(name, is_delete)` 联合唯一不会冲突。

> 例外：为了方便安全审计，以下场景可保留单列唯一约束：
>
> - 用户身份标识（如 `username`），出于安全与审计需要，即使删除也不允许复用。
> - 全局唯一且不可复用的标识（如系统编码、外部平台 ID）。

## 4. 字段类型约定

| 场景                 | 推荐类型    | 说明                                           |
| -------------------- | ----------- | ---------------------------------------------- |
| 自增主键             | INTEGER     | 使用 `PRIMARY KEY AUTOINCREMENT`               |
| 状态/类型枚举        | VARCHAR     | 长度 20~50，具体值由业务枚举定义               |
| 名称/标题            | VARCHAR(100)| 业务名称统一长度，便于 UI 展示与校验           |
| 描述/备注            | VARCHAR(500)| 较长文本描述                                   |
| 大文本/JSON          | TEXT        | 配置、日志、DDL 等                           |
| URL/路径             | VARCHAR(500)| 文件路径、访问地址等                           |
| 时间戳               | INTEGER/BIGINT | 秒级时间戳，具体类型与同表其他字段保持一致 |
| 删除标志             | INTEGER     | 0 或记录 id，由 MyBatis-Plus 管理             |

## 5. 变更管理

- 数据库结构变更统一通过 Flyway 迁移脚本管理。
- 已发布的迁移脚本不可直接修改；如需调整，应新增更高版本的迁移脚本。
- 开发阶段尚未合并的迁移脚本可原地修改，但需通知团队成员清理本地数据库。
