# Service 层设计原则

## 原则一：Service 层只抛异常，不返回 Result

**做什么**：成功路径返回业务数据，失败路径抛 `BusinessException`，`Result` 的组装是 Controller 的职责。

**为什么**：`Result` 是 HTTP 接口层的概念，Service 不应该知道自己最终会被 Controller、定时任务还是另一个 Service 调用。如果 Service 返回 `Result`，调用方就必须写大量 if-else 来判断 code，而且正常返回值没有地方存放（`Result<T>` 的 `data` 和失败状态混在一起）。抛异常的好处是：成功路径代码无噪音，失败路径由 `GlobalExceptionHandler` 统一处理，调用方不需要关心错误分支。

---

## 原则二：参数校验分两层，Service 只做业务规则校验

**做什么**：
- **格式校验**（非空、长度、格式正则）放在 DTO 的 `@Valid` 注解上，Controller 入口拦截，进不了 Service
- **业务规则校验**（用户名是否重复、资源是否存在、状态是否合法）放在 Service 里

**为什么**：格式校验是"输入合法性"问题，跟业务逻辑无关，放在 DTO 层可以复用（同一个 DTO 被多个接口使用时不需要重复校验）。如果放在 Service 里，一是代码冗长（每个方法开头一堆 null check），二是职责不清（Service 变成了校验器 + 业务处理器的混合体），三是测试困难（单元测试 Service 时必须传合法数据才能测到真正的业务分支）。

---

## 原则三：跨模块依赖要确认当前版本真实存在

**做什么**：从旧项目 copy 代码时，凡是依赖其他模块 Service 的地方，必须先确认该模块在当前版本真实存在，不存在的依赖一律先删掉。

**为什么**：旧项目的功能边界和新项目不一样（如旧项目有用户组/多租户，新项目社区版没有），直接 copy 会带入当前版本根本不存在的依赖，代码能写但编译不过，或者运行时找不到 Bean，排查成本远高于一开始就删掉重写。

---

## 原则四：跨方法的重复逻辑提取成私有方法

**做什么**：同一块逻辑（如查询条件构建、字段更新判断）在多个方法里出现时，提取成语义清晰的 `private` 方法。

**为什么**：重复代码的最大问题不是"多写了几行"，而是"改一处时容易漏改其他地方"。比如搜索用户的关键词匹配逻辑出现在三个方法里，有一天要加一个新的匹配字段（如 `phone`），就要改三处，漏掉任何一处都是 bug。私有方法还能让每个业务方法的主干逻辑更清晰——读代码时看到的是 `buildKeywordWrapper(keyword)` 而不是十几行 if-else。

---

## 原则五：常量不硬编码在 Service 里

**做什么**：有业务含义的魔法值（如受保护的用户名 `"admin"`、受保护的用户 ID `1L`）统一放到模块内的 `XxxConstants` 类里，Service 只引用常量名。

**为什么**：硬编码的字面量散落在代码各处有两个问题：一是可读性差（看到 `1L` 不知道是什么含义，看到 `UserConstants.PROTECTED_USER_ID` 立刻明白）；二是修改成本高（如果受保护用户名从 `"admin"` 改成 `"root"`，硬编码需要全局搜索替换，常量只改一处）。
